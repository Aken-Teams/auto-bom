# 罐头匹配 & 标准作业匹配 — 完整说明、问题与待确认清单

> 最后更新：2026-06-24
> 范围：精灵流程第 3 步「配置罐头」的罐头自动匹配；以及第 5 步生成的 sequences「工序 → 标准作业ID」匹配。
> 用途：(1) 记录系统现状与已修问题；(2) 汇整需要跟 USER 确认的规则细节（含实际数据举例，可直接转交 USER 标注）。
> 两处都是同一类问题：**匹配是领域知识，需要 USER 给真实规则**。

---

## 0. 一句话现状

罐头分**焊接 / 成型 / 包装**三类。系统目前：焊接按「原件 WAF code」精确匹配；成型/包装靠「配置罐头」页的规则面板（使用者自填条件）。
**结论先行**：通用的「自填规则」面板无法表达真实的领域匹配逻辑（尤其焊接疑似按 mil 而非 WAF），**需要 USER 提供真实规则**才能做对 —— 见第 5、6 节。

## 1. 罐头档的真实结构（含问题2）

上传的「罐头对照表」每一行 = **一个罐头定义**，类型藏在描述栏（第 11 栏）的前缀：

| 类型 | 描述前缀 | 是否有 WAF | 范例 |
|------|---------|-----------|------|
| 焊接罐 weld | `焊接/…` | 有 | `SMC_DW0163`（绑 `WAF108622_CP`，SKY_55MIL）|
| 成型罐 mold | `成型/…` | 无（通用）| `SMC_MD0015`（Auto Mold）|
| 包装罐 pack | `包裝/…` | 无（通用）| `SMC_PA0009`（TR 7"/0.8K）、`SMC_PA0011`（TR 13"/3K）|

### 问题2（已修）：解析丢行 21 → 18
- **现象**：罐头档实际 21 行，系统只带出 18 行。
- **根因**：旧解析「WAF（B 栏）为空就跳过」，把 3 个**无 WAF 的通用罐头**（1 成型 + 2 包装）全丢了。
- **修复**：改为逐行按描述前缀分类，全部读入（18 焊接 + 3 通用）。已端到端验证：上传回传 `count=21 / weld_count=18`，通用罐头 3 个。

## 2. 目前系统的匹配逻辑

### 焊接罐（weld）
- 按料号的**原件 WAF code 一对一精确匹配**罐头档（`CanTemplate`，键 = `waf_code`）。
- ⚠️ 这条逻辑**可能是错的** —— 见问题3（第 5 节）。

### 成型 / 包装罐（mold / pack）— 通用罐头
- 不绑 WAF，靠「配置罐头」页的**规则面板**指派。
- 候选罐头**从当前上传的罐头档自动带出**（非写死）。

---

## 3. 规则面板与规则模型

一条规则 = **`字段` + `运算子` + `值`  →  `目标罐头`**

- **字段**：料号 `item_no` / TYPE / FAMILY / PACKAGE / 原件 `component`
- **运算子**：

  | 运算子 | 含义 |
  |--------|------|
  | 全部 all | 无条件套用（兜底，放最后）|
  | 包含 contains | 值是字段的子字符串 |
  | 等于 equals | 完全相等 |
  | 正则 regex | Python `re.search(值, 字段)`，区分大小写、部分匹配；例 `_R[12]_`=R1或R2、`^MB`=以 MB 开头；语法错误视为不匹配 |

- **优先级**：同一类罐头，规则**由上到下逐条试，第一条命中即套用并停止**（顺序＝优先级）。
- **预设种子**（每次进配置罐头从当前档案带出）：成型单个→「全部→该罐头」；包装两个→预填「料号含 `_R1_`→PA0009」「料号含 `_R2_`→PA0011」当讨论起点。

### 面板已有的辅助功能
- **每栏筛选**（Excel 式）：料号/原件/替代结构用文字框；焊接/成型/包装用下拉（全部 / **（未匹配）** / 具体值）。可快速筛出未匹配的料号。
- 正则输入框有 placeholder/tooltip 提示语法。

---

## 4. 设计决定

- **规则不存数据库**：每次进「配置罐头」从当前罐头档重新带出预设；按「自动匹配」只套到本次任务，不持久化。
  - 原因：流程是「每次新的底稿＋罐头档 → 筛选 → 配置」，规则可能每次不同，存库反而难厘清。
  - `can_rules` 表/接口骨架保留但未启用，未来若要「记住/复用规则」再开。
- **匹配会覆盖对应栏**：命中规则即写入；手动改过的值若再按一次自动匹配且命中会被覆盖。建议「先自动匹配，再手动调例外」。

## 5. 已发现的问题

### 问题2：解析丢行（已修）
见第 1 节。

### 问题3：焊接罐匹配键可能不是 WAF（未决，需 USER 确认）
- **现象**：USER 回报「焊接罐头料号识别为空」，约 **267/372** 笔焊接未匹配。
- **实例**：

  | 料号 | 原件(WAF) | 替代结构 | 现在结果 | USER 期望对到 |
  |------|-----------|---------|---------|--------------|
  | MB55_R1_00001 | WAF108425_CP | ACP_**68** | 空白（档里无此 WAF）| WAF109296_CP（**68mil**, SKY, 焊接罐 SMC_DW0164）|

- **已验证**：`WAF108425_CP` 确实不在罐头档焊接清单内；档内有一个 `mil=68` 的焊接罐 `WAF109296_CP`。
- **推断**：焊接真正的匹配键是 **mil 值（由替代结构 `ACP_68 → 68`）**，而非原件 WAF。这正是「未匹配 267」的根因 —— 现行用 WAF 精确匹配，凡原件 WAF 不在档内（仅 ~28 个）的料号全空。
- **复杂点**：同一 mil 可能有多个焊接罐候选（档内 70mil、100mil 各有 2 个，function 不同），光靠 mil 无法唯一确定，**很可能是 mil + function**。

### 问题4：sequences 工序 → 标准作业ID 匹配缺口（未决，需 USER 确认）
- **现象**：生成的 `sequences-raw` 里 `STANDARD_OPERATION_ID` 栏，12 道工序只填了 3 道、且都是同一个错值，其余 9 道空白。
- **已验证**：WXBMR004 档案正常（24736 行有效）。现行逻辑把 WXBMR004 某栏当工序号，但该栏只有 10/20/30/0 四种值（不是工序 10~90），且只取「第一笔」op_id，所以只填 seq 10/20/30、值还相同（679347 = 第一笔 TVS TGPP-70）。
- **范本要求**：12 道工序（切割/焊接/成型/切脚/Burning/外包前/贝维特/佰润/欣捷/外包后/TMTT/FQC）**各有不同 op_id**，备注写明「**切割按晶片大小、焊接~TMTT 按 family**」。
- **结论**：占位实现，非本次改动、非档案问题。需 USER 的工序匹配规则才能做对。
- **🔑 已从 `data/` 范本逆推出规则（2026-06-24）**：用范本输出 `sequences-raw 3.3.xlsx`（= 对答案）反查 `WXBMR004` 与底稿，料号 `ER3A_R1_00001`（family=ESM-30H、替代结构 ACP_70）的 12 道工序结果：
  - WXBMR004 结构：`col1=标准作业ID`、`col3=摘要(产品码)`、`col4=部门(=工序)`、`col6=子序号(10/20/30，非工序)`。
  - **焊接/成型/切脚/Burning/外包后/TMTT/FQC**：摘要 = **`{family} {工序} {BOP}M`**（如 `ESM-30H 焊接 ACPM`；family=底稿 family，BOP=替代结构前3位）。
  - **电镀（贝维特/佰润/欣捷）**：摘要 = `{package} 電鍍-{厂别}/{package}/5um`，**按 package（SMC）匹配**。
  - **外包前**：摘要固定 `外包前`（与料号无关）。
  - **切割**：摘要 = `ERG EGPP-70` 之类，**按晶片（function/supplier + mil）**——唯一还要确认确切来源字段。
  - 注意部门用繁体（切腳/貝維特/外包後），匹配时要做繁简归一。
  - **验证方式**：`data/` 的 3 个范本输出即对答案，可重现底稿→比对。
- **✅ 已实现并验证（2026-06-24）**：`bom_generator.build_std_op_index` + `resolve_std_op` + 改写 `generate_sequences`。
  - 用 `data/` 范本（C-CMAX导入清单3-3(1) + WXBMR004 + 罐头）整档回归比对答案：
    - **排除切割（暂留空）→ 99.7% 正确**（1991 格仅 6 笔电镀边角不符）。
    - 全部 12 工序：正确 91.4%、空白 181（主要切割）、错误 6。
  - **仍待 USER**：
    1. **切割**：晶片码（如 `ERG EGPP-70`）是 **TYPE 级对照**——同 family/原件、不同 TYPE 会对到不同晶片（FR3A→ERG、FR3D→UFG），数据推不出，**现暂留空**。需 USER 给「TYPE/料号 → 晶片码或 op_id」对照表。
    2. **外包前**：答案用「通用外包前」代码最大者（669332），非按封装的「SMC 外包前」；规则待确认。
    3. **电镀 6 笔边角**：少数 package/厂别变体未对上，待个案确认。

### 问题5：解析按「栏位位置」而非「表头名」，移动栏位会静默读错（限制，需 USER 知悉）
- **现象/风险**：所有 Excel 解析都按**固定栏位位置**硬编码读取（非按表头名）。若上传档把某栏移动、或插入/删除栏位，后面所有栏位错位，系统会**照旧位置读到错数据**，**且不报错**，照样产出错误结果——比报错更难发现。
- **硬编码位置**（`excel_parser.py`）：
  - BOM底稿：料号=第1栏、摘要=2、替代结构=6、原件WAF=14、原件摘要=15
  - 罐头档：WAF=2、罐头代码=11、描述=12
  - 标准作业：op_id=1、seq=6
- **两个方向**：(1) 约定上传档栏位顺序/数量不可变（最简单）；(2) 改成按表头名匹配（更稳，但要先定每栏标准表头名、且表头不可改名）。
